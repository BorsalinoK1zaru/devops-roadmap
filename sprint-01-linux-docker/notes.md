## День 2. Теория

PowerShell — оболочка Windows. В ней я могу запускать wsl, docker, git и Windows-команды.

WSL — Linux-среда внутри Windows. В ней я выполняю Linux-команды: uname, ls, cat, grep, chmod, apt.

Docker container — изолированная среда для запуска приложения. Контейнер создаётся из образа.

Docker image — шаблон, из которого создаются контейнеры.

docker run создаёт и запускает контейнер.

Проброс портов 8080:80 означает: порт 8080 на моём компьютере перенаправляется на порт 80 внутри контейнера.

docker ps показывает запущенные контейнеры.

docker logs показывает логи контейнера.

docker exec позволяет выполнить команду внутри контейнера.

docker stop останавливает контейнер.

docker rm удаляет контейнер.

## День 2. Что я понял

1. Чем PowerShell отличается от WSL? - PowerShell - обололчка ОС Windows, где я пишу комманды, WSL - Linux
2. Где выполняется команда `uname`? - WSL
3. Что делает команда `docker run -d -p 8080:80 nginx`? - создаю и запускаю контейнер в фоне на порте 8080 (на моей машине), и прослушиваю порт 80 внутри контейнера  
4. Что означает `8080:80`? 8080 - порт моей машины, 80 - порт контейнера, который я собираюсь прослушивать
5. Чем контейнер отличается от виртуальной машины? Виртуальная машина запускает полноценную отдельную ОС со своим ядром. Контейнер не запускает отдельную ОС, а изолирует приложение и его окружение, используя ядро хостовой системы. Контейнер легче и быстрее VM.
6. Как зайти внутрь контейнера? - docker exec -it devops-nginx sh
7. Где лежит стандартная HTML-страница Nginx внутри контейнера? /usr/share/nginx/html/index.html

## День 3. Что я понял

- Docker image — это шаблон для создания контейнера.
- Docker container — это запущенный экземпляр image.
- Volume mount позволяет подключить папку с моей машины внутрь контейнера.
- В команде `-v ${PWD}:/usr/share/nginx/html` моя текущая папка заменяет стандартную папку сайта Nginx внутри контейнера.

## День 4. Что я понял

- Dockerfile — это инструкция для сборки Docker image.
- `FROM nginx:latest` берёт готовый Nginx image как основу.
- `COPY index.html /usr/share/nginx/html/index.html` копирует мой HTML-файл внутрь image.
- `docker build` собирает image.
- `docker run` запускает container из image.
- Tag нужен, чтобы различать версии image, например `my-nginx-site:day4`.
- Точка в `docker build -t my-nginx-site:day4 .` означает текущую папку как build context.

## День 5. Что я понял

- Docker image состоит из слоёв.
- Инструкции `FROM`, `LABEL`, `RUN`, `COPY` участвуют в сборке image.
- Docker cache позволяет не пересобирать неизменённые шаги.
- Если файлы не менялись, повторный `docker build` проходит быстрее.
- Если изменить `index.html`, слой с `COPY index.html ...` должен быть пересобран.
- `docker history` показывает историю слоёв image.
- Порядок инструкций в Dockerfile важен: редко меняющиеся шаги лучше ставить выше, часто меняющиеся — ниже.

## День 5. Уточнения

- Docker cache — это сохранённые результаты предыдущих шагов сборки, которые Docker хранит локально и переиспользует при повторном `docker build`.
- После изменения локального `index.html` нужно пересобрать image, потому что файл внутри image был зафиксирован на момент предыдущей сборки.
- `docker build` собирает image из Dockerfile.
- `docker run` создаёт и запускает container из уже готового image.

## День 6. Что я понял

- Flask позволяет создать простое Python web-приложение.
- `requirements.txt` хранит зависимости Python-проекта.
- `WORKDIR /app` задаёт рабочую папку внутри image.
- `RUN pip install ...` устанавливает зависимости во время сборки image.
- `COPY app.py .` копирует код приложения внутрь image.
- `EXPOSE 5000` документирует порт приложения, но не открывает его наружу.
- `docker run -p 5000:5000` пробрасывает порт хоста в контейнер.
- Gunicorn запускает Flask-приложение внутри контейнера.

## День 6. Уточнения

- `RUN` выполняется во время `docker build` и изменяет image.
- `CMD` выполняется при запуске container через `docker run`.
- `-p 5000:5000` означает: порт 5000 на моей машине перенаправляется на порт 5000 внутри контейнера.
- Левый порт в `-p` — порт хоста, правый порт — порт контейнера.
- Приложение внутри контейнера слушает порт, указанный в Gunicorn: `--bind 0.0.0.0:5000`.

## День 7. Что я понял

- Переменные окружения позволяют менять настройки приложения без изменения кода.
- В Python переменные окружения можно читать через `os.getenv()`.
- `os.getenv("APP_VERSION", "dev")` означает: взять `APP_VERSION`, а если её нет — использовать `dev`.
- В Docker переменные можно передавать через `docker run -e`.
- `-p 5050:5000` означает: снаружи приложение доступно на порту `5050`, а внутри контейнера слушает `5000`.
- Один и тот же image можно запускать с разными переменными окружения.
- `os.getenv("APP_VERSION", "dev")` берёт значение `APP_VERSION` из окружения, а если переменной нет — использует значение по умолчанию `dev`.

## День 8. Что я понял

- Docker network позволяет контейнерам общаться друг с другом.
- `localhost` внутри контейнера означает сам контейнер, а не соседний контейнер.
- В custom Docker network контейнеры могут обращаться друг к другу по имени.
- `--network devops-net` подключает контейнер к сети `devops-net`.
- Redis не обязательно пробрасывать наружу через `-p`, если к нему обращается только другой контейнер.
- `REDIS_HOST=day8-redis` означает, что Python-приложение ищет Redis по имени контейнера.
- `docker network inspect` показывает, какие контейнеры подключены к сети.

 ## День 9. Что я понял

- Контейнер можно удалить, и его внутренняя файловая система исчезнет.
- Docker volume хранит данные отдельно от контейнера.
- `-v redis-data:/data` подключает volume `redis-data` в папку `/data` внутри Redis.
- Если новый Redis container запустить с тем же volume, он может использовать сохранённые данные.
- Redis может хранить данные в памяти, поэтому для надёжного теста можно выполнить `redis-cli SAVE`.
- `docker volume ls` показывает volumes.
- `docker volume inspect` показывает информацию о конкретном volume.

## День 9. Уточнение

- `app.py` не запускает два контейнера. Он находится только внутри Python-контейнера.
- Redis запускается отдельным контейнером из image `redis:7-alpine`.
- Python-приложение подключается к Redis по имени контейнера через Docker network.
- Docker запускает контейнеры, Docker network связывает их, а Docker volume хранит данные Redis.

## День 10. Что я понял

- Docker Compose позволяет описать несколько контейнеров в одном YAML-файле.
- `services` — список сервисов приложения.
- `build: .` означает собрать image из Dockerfile в текущей папке.
- `image` задаёт имя image.
- `ports` работает как `-p` в `docker run`.
- `environment` работает как `-e` в `docker run`.
- `volumes` работает как `-v` в `docker run`.
- Compose автоматически создаёт network для проекта.
- В Compose сервисы могут обращаться друг к другу по service name, например `redis`.
- `docker compose up -d` запускает проект в фоне.
- `docker compose down` останавливает и удаляет контейнеры проекта и сеть, но не удаляет named volumes без `-v`.

## День 10. Уточнения

- В `REDIS_HOST` мы указываем `redis`, потому что это service name в `compose.yaml`.
- Docker Compose создаёт внутреннюю сеть, где сервисы могут обращаться друг к другу по имени service.
- IP контейнера может измениться после пересоздания, поэтому лучше использовать service name.
- `docker compose down` удаляет контейнеры и сеть проекта, но не удаляет named volumes без флага `-v`.

## День 11. Что я понял

- `.env` позволяет хранить настройки проекта отдельно от `compose.yaml`.
- `${APP_PORT}:5000` подставляет внешний порт из `.env`.
- `env_file` передаёт переменные из файла внутрь контейнера.
- `healthcheck` проверяет, что сервис не просто запущен, а реально работает.
- Для Redis можно использовать проверку `redis-cli ping`.
- `depends_on` с `condition: service_healthy` заставляет app ждать healthy-статус Redis.
- `docker compose ps` показывает статус сервисов, включая health.
